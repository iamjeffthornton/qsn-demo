LangGraph
Run your live LangGraph agent
Copy the agent files, run the pipeline, and watch 5 agents work in sequence with real Claude API calls.
What's happening under the hood
When you type "python qsn_agent.py", LangGraph wakes up your 5-agent assembly line. It fires Router first, waits for the answer, passes the updated notebook to Analyst, and so on down the chain. Every agent calls the real Claude API and you pay only for the tokens used.
2A Copy the agent files into your project folder ›

Your project folder should look exactly like this. The files were created alongside this guide — copy them in now.

qsn-demo/ ├── qsn_agent.py ← the main agent (5 nodes + graph) ├── requirements.txt ← Python packages needed ├── Dockerfile ← recipe to containerize your agent ├── docker-compose.yml ← run with one command locally └── k8s/ ├── deployment.yaml ← K8s: run 3 replicas of the agent └── service.yaml ← K8s: expose agent to traffic

3
Docker
Package your agent in a Docker container
Turn your Python script into a portable, self-contained container that runs identically on your laptop, Azure, or any cloud.
Why this matters for the interview
Right now your agent only works on your laptop, in your virtual environment, with your Python version. Docker packages your code AND its entire environment into one box. That box runs anywhere. Accenture deploys agents for dozens of clients — Docker is how they ensure "it works on my machine" becomes "it works everywhere."
3A Build your Docker image ›

The Dockerfile in your project folder is already written. This command reads it and builds your image. It will take about 2-3 minutes the first time (downloading base Python image). Subsequent builds are much faster because Docker caches layers.

Terminal — inside qsn-demo/
docker build -t qsn-agent:1.0 . # -t qsn-agent:1.0 = name it "qsn-agent" version "1.0" # . = use the Dockerfile in this folder Step 1/9 : FROM python:3.11-slim AS builder ---> Pulling from library/python Step 2/9 : WORKDIR /build Step 3/9 : COPY requirements.txt . Step 4/9 : RUN pip install --no-cache-dir... Installing collected packages: langgraph, anthropic... Step 5/9 : FROM python:3.11-slim AS runtime Step 6/9 : COPY --from=builder /install /usr/local Step 7/9 : COPY qsn_agent.py . Step 8/9 : RUN useradd qsnuser Step 9/9 : CMD ["python", "qsn_agent.py", "--health"] Successfully built a3f4b8c9d2e1 Successfully tagged qsn-agent:1.0
3B Run your agent inside Docker ›

This runs your agent inside a container. The -e flag injects your API key as an environment variable — exactly how Kubernetes will do it in production.

Terminal
docker run \ -e ANTHROPIC_API_KEY=$ANTHROPIC_API_KEY \ qsn-agent:1.0 # Same output as running python directly — but now it's in Docker! 💚 Health check server running on :8080/health 🏉 QSN RUGBY CONTENT FACTORY — LANGGRAPH LIVE DEMO 🏉 [1/5] ROUTER AGENT — deciding content type...

The moment it works inside Docker: You've just proven your agent is portable. That same command works on any server in the world that has Docker installed.

3C Essential Docker commands you must own ›

These 10 commands cover 90% of everything you'll do with Docker. Practice them until they're automatic.

Docker Cheat Sheet
# BUILD — create an image from a Dockerfile docker build -t myapp:v1 . # RUN — start a container from an image docker run myapp:v1 docker run -d myapp:v1 # -d = run in background docker run -p 8080:8080 myapp:v1 # -p = map ports # LIST — see what's running docker ps # running containers docker images # all images on your machine # LOGS — see what a container printed docker logs container-name docker logs -f container-name # -f = follow (live stream) # EXEC — open a shell inside a running container docker exec -it container-name /bin/bash # like SSH into a container # STOP / REMOVE docker stop container-name docker rm container-name docker rmi qsn-agent:1.0 # remove an image
3D Run with docker-compose (simpler for local dev) ›

docker-compose reads your docker-compose.yml file and starts everything with one command. Easier than long docker run commands.

Terminal
docker-compose up # start everything docker-compose up -d # start in background docker-compose down # stop everything docker-compose logs -f # watch all logs live

4
Kubernetes
Deploy your agent to a local Kubernetes cluster
Create a K8s cluster on your laptop, deploy 3 replicas of your agent, simulate a crash, and watch K8s auto-heal it.
The moment that makes K8s click
Deploy 3 replicas of your agent. Then manually delete one pod. Watch Kubernetes instantly spin up a replacement — without you doing anything. That is the magic of K8s. In production, if an agent crashes at 3am, K8s restarts it before anyone wakes up.
4A Create your local Kubernetes cluster ›

One command creates a full K8s cluster inside Docker on your laptop. k3d is lightweight — it won't slow your machine down.

Terminal
k3d cluster create qsn-cluster --agents 2 # Creates a cluster named "qsn-cluster" with 2 worker nodes INFO Creating cluster 'qsn-cluster' INFO Creating network 'k3d-qsn-cluster' INFO Creating node 'k3d-qsn-cluster-server-0' INFO Creating node 'k3d-qsn-cluster-agent-0' INFO Creating node 'k3d-qsn-cluster-agent-1' INFO Cluster 'qsn-cluster' created successfully! # Verify cluster is running: kubectl get nodes NAME STATUS ROLES AGE k3d-qsn-cluster-server-0 Ready master 30s k3d-qsn-cluster-agent-0 Ready worker 20s k3d-qsn-cluster-agent-1 Ready worker 20s
4B Load your Docker image into the cluster ›

k3d clusters can't pull from your local Docker by default. This command imports your image into the cluster so the nodes can use it.

Terminal
k3d image import qsn-agent:1.0 -c qsn-cluster INFO Importing image(s) into cluster 'qsn-cluster' INFO Successfully imported image(s) # In production (Azure), you'd push to ACR instead: # docker push myregistry.azurecr.io/qsn-agent:1.0
4C Create the API key Secret ›

Never put your API key in code or YAML files. Create a K8s Secret — an encrypted key-value store. Your pods read it at runtime.

Terminal
kubectl create secret generic qsn-secrets \ --from-literal=anthropic-api-key=$ANTHROPIC_API_KEY secret/qsn-secrets created # Verify it exists (values are hidden — base64 encoded): kubectl get secret qsn-secrets NAME TYPE DATA AGE qsn-secrets Opaque 1 5s
4D Deploy your agent — 3 replicas running ›

Apply both YAML files. K8s reads them and creates 3 agent pods plus a Service to route traffic to them.

Terminal — THE BIG MOMENT
kubectl apply -f k8s/ deployment.apps/qsn-agent created service/qsn-agent-service created # Watch pods start up in real time (-w = watch) kubectl get pods -w NAME READY STATUS RESTARTS qsn-agent-7d4f8b9c6-x2k8p 0/1 ContainerCreating 0 qsn-agent-7d4f8b9c6-m9n3q 0/1 ContainerCreating 0 qsn-agent-7d4f8b9c6-p4r7t 0/1 ContainerCreating 0 qsn-agent-7d4f8b9c6-x2k8p 1/1 Running 0 qsn-agent-7d4f8b9c6-m9n3q 1/1 Running 0 qsn-agent-7d4f8b9c6-p4r7t 1/1 Running 0 # 3 pods running! Press Ctrl+C to stop watching
What 1/1 Running means
"1/1" = 1 container out of 1 is running. "Running" = it's alive and healthy. Three of these = 3 copies of your agent, all live, all ready to handle requests.
4E Simulate a crash — watch K8s auto-heal ✨ ›

This is the moment that makes K8s click. Delete one pod manually. K8s detects the replica count dropped below 3 and immediately creates a replacement. This is why we run 3 replicas — one crashes, two keep running, K8s starts a third.

Terminal — open 2 tabs for this one
# TAB 1: Watch pods continuously kubectl get pods -w # TAB 2: Kill one pod by name (copy a pod name from Tab 1) kubectl delete pod qsn-agent-7d4f8b9c6-x2k8p pod "qsn-agent-7d4f8b9c6-x2k8p" deleted # Back in Tab 1 — watch this happen automatically: qsn-agent-7d4f8b9c6-x2k8p 1/1 Terminating 0 qsn-agent-7d4f8b9c6-NEW45 0/1 Pending 0 ← new one! qsn-agent-7d4f8b9c6-NEW45 1/1 Running 0 ← healed! # The new pod was created and running in ~15 seconds # WITHOUT you doing anything. That's Kubernetes.
4F Core kubectl commands — practice these daily ›
kubectl Cheat Sheet
# SEE WHAT'S RUNNING kubectl get pods # list all pods kubectl get pods -w # watch live updates kubectl get deployments # see all deployments kubectl get services # see all services kubectl get all # see everything at once # INSPECT A POD kubectl describe pod pod-name # detailed info + events kubectl logs pod-name # what the pod printed kubectl logs -f pod-name # follow logs live # DEPLOY / UPDATE kubectl apply -f k8s/ # deploy or update resources kubectl rollout status deployment/qsn-agent # watch rollout kubectl rollout undo deployment/qsn-agent # rollback! # SCALE kubectl scale deployment qsn-agent --replicas=5 # 5 pods # CLEAN UP kubectl delete -f k8s/ # remove everything you deployed k3d cluster delete qsn-cluster # delete the whole cluster
Interview power move

"I deployed 3 replicas of the containerized agent to a local k3d cluster and tested the self-healing behavior by manually deleting a pod. K8s brought the replacement up in under 15 seconds. For production AKS, the same deployment.yaml works — just change the image tag to point to Azure Container Registry. I also configured liveness and readiness probes so K8s knows when an agent is truly healthy vs just started."

5
CI/CD
Automate deployment with GitHub Actions
Push code to GitHub → pipeline runs tests → builds Docker image → deploys automatically. Zero manual steps.
CI/CD = the QSN publishing workflow, but for code
When you finish a QSN video, it goes through editing → review → approval → publish. CI/CD does this for code automatically. Every time you push to GitHub, the pipeline: runs tests (editing), builds Docker (review), deploys (publish). If tests fail, the pipeline stops. Nothing broken reaches production.
5A Push your project to GitHub ›

Create a GitHub repo and push your project. The .github/workflows/deploy.yml file is already in your project — GitHub Actions will auto-detect it.

Terminal
# Initialize git and push to GitHub git init git add . git commit -m "initial: QSN LangGraph agent" # Create repo at github.com, then push: git remote add origin https://github.com/YOUR-USERNAME/qsn-demo.git git push -u origin main # IMPORTANT: Add your API key as a GitHub Secret # Go to: GitHub repo → Settings → Secrets → Actions → New secret # Name: ANTHROPIC_API_KEY # Value: your actual key
5B Watch the pipeline run ›

After you push, go to your GitHub repo → Actions tab. You'll see the pipeline running. Green checkmarks = passing. Red X = failed with a log showing exactly what broke.

GitHub Actions — what you'll see
GitHub → Your Repo → Actions tab ✓ Run Tests Set up Python 3.11 3s Install dependencies 45s Run tests 2s ✅ Import OK ✅ Graph compiled OK ✓ Build Docker Image Checkout code 2s Build Docker image 2m 14s Test Docker image 8s ✅ Health check passed ✓ Deploy to Kubernetes ✅ Deployment complete Pipeline passed in 3m 28s
5C Trigger a deployment by changing code ›

Change one line of code and push. Watch the pipeline automatically rebuild and redeploy. This is the point — you never manually deploy again.

Terminal — trigger a new deploy
# Make any change to qsn_agent.py # e.g., update the Analyst agent's system prompt git add . git commit -m "feat: update analyst agent prompt" git push # GitHub Actions automatically: # 1. Runs tests on your new code # 2. Builds a new Docker image tagged with the commit SHA # 3. Does a rolling update on K8s (zero downtime) # 4. Old pods shut down AFTER new pods are healthy
Interview power move

"My CI/CD pipeline runs on every push to main. Tests first — if they fail, nothing deploys. Then Docker build tagged with the commit SHA so I can trace any production pod back to the exact code that built it. Rolling update strategy means zero downtime — new pods come up before old ones go down. I can show you the GitHub Actions workflow file right now."
